# UC-1: Аутентификация

## TC-1.1 Успешная регистрация

**FR:** FR-1, FR-2

**Предусловие:** пользователь не зарегистрирован.

**Шаги:**

1. На SPA нажать кнопку **«Зарегистрироваться»**.
2. Убедиться, что произошел редирект на страницу авторизации Zitadel.
3. Заполнить регистрационную форму:
   - Email;
   - Display Name;
   - Username;
   - Password.
4. Подтвердить регистрацию.
5. Убедиться, что произошел редирект обратно в SPA.

**Ожидаемый результат:**

- Аккаунт успешно создан в Zitadel.
- Пользователь автоматически авторизован.
- Пользователь перенаправлен на главную страницу приложения.
- В интерфейсе отображаются имя и username пользователя.

---

## TC-1.2 Регистрация с уже существующим email

**FR:** FR-1

**Предусловие:** пользователь с указанным email уже существует.

**Шаги:**

1. На SPA нажать кнопку **«Зарегистрироваться»**.
2. Убедиться, что произошел редирект на страницу Zitadel.
3. Заполнить все поля формы, указав email уже существующего аккаунта.
4. Подтвердить регистрацию.

**Ожидаемый результат:**

- Zitadel отображает сообщение об ошибке - пользователь с таким email уже существует.
- Новый аккаунт не создается.
- Редирект обратно в SPA не выполняется.

---

## TC-1.3 Успешный вход

**FR:** FR-2

**Предусловие:** пользователь зарегистрирован.

**Шаги:**

1. На SPA нажать кнопку **«Войти»**.
2. Убедиться, что произошел редирект на страницу Zitadel.
3. Ввести email и корректный пароль.
4. Подтвердить вход.
5. Убедиться, что произошел редирект обратно в SPA.

**Ожидаемый результат:**

- Пользователь успешно авторизован.
- Пользователь перенаправлен на главную страницу приложения.
- В интерфейсе отображаются имя и username пользователя.

---

## TC-1.4 Вход с неверным паролем

**FR:** FR-2

**Предусловие:** пользователь зарегистрирован.

**Шаги:**

1. На SPA нажать кнопку **«Войти»**.
2. Дождаться перехода на страницу Zitadel.
3. Ввести корректный email и неверный пароль.
4. Подтвердить вход.

**Ожидаемый результат:**

- Zitadel отображает ошибку авторизации.
- Пользователь остается на странице входа Zitadel.
- Редирект обратно в SPA не выполняется.

---

## TC-1.5 Silent Token Refresh

**FR:** FR-3

**Предусловие:** пользователь авторизован, сессия Zitadel активна.

**Шаги:**

1. Выполнить вход в приложение.
2. Дождаться истечения срока действия access token.
3. Продолжить работу в приложении без обновления страницы.

**Ожидаемый результат:**

- Access token автоматически обновляется в фоновом режиме через скрытый iframe.
- Пользователь остается авторизованным.
- Редирект на страницу Zitadel не происходит.
- Текущая сессия не прерывается.

# UC-2: Управление сервером

## TC-2.1 Успешное создание сервера

**FR:** FR-5

**Предусловие:** пользователь авторизован.

**Шаги:**

1. Нажать **«Создать сервер»**.
2. Ввести название сервера.
3. Подтвердить создание.

**Ожидаемый результат:**

- Сервер успешно создан.
- Создателю автоматически назначается роль **Owner**.
- Автоматически создается текстовый канал `general`.

---

## TC-2.2 Создание текстового канала

**FR:** FR-6

**Предусловие:** пользователь авторизован и имеет роль **Owner** или **Admin** на сервере.

**Шаги:**

1. Открыть сервер.
2. Нажать **«+»** рядом с разделом каналов.
3. Ввести название канала, выбрать тип **text**.
4. Подтвердить создание.

**Ожидаемый результат:**

- Новый текстовый канал отображается в списке каналов у всех участников без перезагрузки страницы.
- Остальные участники получают событие `channel.created` по SignalR.

---

## TC-2.3 Создание голосового канала

**FR:** FR-7

**Предусловие:** пользователь авторизован и имеет роль **Owner** или **Admin** на сервере.

**Шаги:**

1. Открыть сервер.
2. Нажать **«+»** рядом с разделом каналов.
3. Ввести название канала, выбрать тип **voice**.
4. Подтвердить создание.

**Ожидаемый результат:**

- Новый голосовой канал отображается в списке каналов у всех участников без перезагрузки страницы.
- Остальные участники получают событие `channel.created` по SignalR.

---

## TC-2.4 Удаление канала

**FR:** FR-10

**Предусловие:** пользователь авторизован и имеет роль **Owner** или **Admin** на сервере; канал существует.

**Шаги:**

1. Открыть сервер.
2. ПКМ на канале → **«Удалить канал»**.
3. Подтвердить удаление.

**Ожидаемый результат:**

- Guild Service возвращает `204 No Content`.
- Канал исчезает из списка у всех участников без перезагрузки.
- Остальные участники получают событие `channel.deleted` по SignalR.
- Если удалялся голосовой канал - все участники принудительно отключаются от него.

---

## TC-2.5 Member пытается удалить канал через API

**FR:** FR-10

**Предусловие:** пользователь авторизован и имеет роль **Member** на сервере.

**Шаги:**

1. Отправить API-запрос `DELETE /api/guilds/{guildId}/channels/{channelId}` с JWT.

**Ожидаемый результат:**

- Guild Service возвращает `HTTP 403 Forbidden`.
- Канал не удаляется.

---

## TC-2.6 Генерация инвайт-ссылки

**FR:** FR-11

**Предусловие:** пользователь авторизован и имеет роль **Owner** или **Admin** на сервере.

**Шаги:**

1. Открыть настройки сервера.
2. Указать TTL ссылки (опционально; `null` - бессрочная).
3. Указать лимит использований (опционально; `null` - без лимита).
4. Нажать **«Создать инвайт»**.

**Ожидаемый результат:**

- Guild Service возвращает `201 Created` с полями `code`, `expiresAt`, `maxUses`.
- Сгенерирована уникальная инвайт-ссылка с заданными параметрами.
- Ссылка доступна для копирования.

---

## TC-2.7 Переход по валидной инвайт-ссылке

**FR:** FR-12

**Предусловие:** пользователь авторизован; ссылка действующая (не истекла, лимит не исчерпан); пользователь не состоит на сервере и не забанен.

**Шаги:**

1. Перейти по действующей инвайт-ссылке.

**Ожидаемый результат:**

- Guild Service возвращает `200 OK` с данными сервера.
- Новому участнику назначается роль **Member**.
- Сервер появляется в списке серверов пользователя.
- Онлайн-участники сервера получают событие `member.joined` по SignalR.

---

## TC-2.8 Использование просроченной инвайт-ссылки

**FR:** FR-12

**Предусловие:** срок действия ссылки истек.

**Шаги:**

1. Перейти по просроченной ссылке.

**Ожидаемый результат:**

- Guild Service возвращает ошибку о том, что ссылка недействительна.
- Пользователь не добавляется на сервер.

---

## TC-2.9 Превышен лимит использований инвайта

**FR:** FR-12

**Предусловие:** лимит использований ссылки исчерпан.

**Шаги:**

1. Перейти по ссылке после достижения лимита.

**Ожидаемый результат:**

- Guild Service возвращает ошибку о недействительности ссылки.
- Пользователь не добавляется на сервер.

---

## TC-2.10 Назначение роли Admin участнику

**FR:** FR-13

**Предусловие:** действие выполняет **Owner**; целевой пользователь имеет роль **Member** на сервере.

**Шаги:**

1. ПКМ на участнике → **«Назначить роль»** → выбрать **Admin**.
2. Подтвердить.

**Ожидаемый результат:**

- Guild Service возвращает `200 OK` с обновленной ролью.
- Все участники сервера получают событие `role.assigned` по SignalR.
- Роль участника обновляется в интерфейсе без перезагрузки.

---

## TC-2.11 Удаление сервера

**FR:** FR-30

**Предусловие:** пользователь авторизован и имеет роль **Owner** на сервере.

**Шаги:**

1. Открыть настройки сервера → **«Удалить сервер»**.
2. Подтвердить удаление (ввести название сервера).

**Ожидаемый результат:**

- Guild Service возвращает `204 No Content`.
- Все участники получают событие `guild.deleted` по SignalR.
- Сервер исчезает из панели у всех участников без перезагрузки.
- Все голосовые каналы закрываются - участники принудительно отключаются от голоса.

# UC-3: Сообщения

**FR:** FR-18

**Шаги:**

1. Пользователь A отправляет сообщение в канал.
2. Пользователь B наблюдает за списком сообщений канала.

**Ожидаемый результат:**

- Новое сообщение появляется у пользователя B без перезагрузки страницы.

---

**Предусловие:** пользователь авторизован и находится в текстовом канале.

**Шаги:**

1. Попытаться отправить пустую строку.

**Ожидаемый результат:**

- Сообщение не отправляется.
- Отображается ошибка валидации.

---

## TC-3.4 Потеря соединения WebSocket

**FR:** FR-18

**Предусловие:** пользователь авторизован и подключен к WebSocket.

**Шаги:**

1. Отключить сеть.
2. Попытаться отправить сообщение.

**Ожидаемый результат:**

- Сообщение не отправляется.
- Пользователь получает уведомление об ошибке соединения.

---

## TC-3.5 Idempotency при отправке сообщения

**FR:** FR-20

**Предусловие:** пользователь авторизован и находится в текстовом канале.

**Шаги:**

1. Вызвать SignalR-метод `SendMessage(channelId, content, idempotencyKey)` с фиксированным значением `idempotencyKey`.
2. Повторно вызвать `SendMessage` с теми же параметрами (тот же `idempotencyKey`).

**Ожидаемый результат:**

- В базе данных создается только одна запись сообщения.
- В канале отображается только одно сообщение.
- Повторный вызов возвращает `200 OK` с данными первого сообщения (без дубля).

---

## TC-3.6 Cursor pagination истории сообщений

**FR:** FR-16

**Предусловие:** пользователь авторизован; в канале есть более 50 сообщений.

**Шаги:**

1. Открыть канал (загружаются последние 50 сообщений).
2. Прокрутить историю сообщений вверх до конца видимой части.

**Ожидаемый результат:**

- Следующая порция сообщений подгружается автоматически (до 50 штук).
- Перезагрузка страницы не требуется.
- Уже загруженные сообщения не дублируются.
- При достижении начала истории подгрузка прекращается.

---

# UC-5: Модерация

## TC-5.1 Кик участника

**FR:** FR-25

**Предусловие:** действие выполняет **Owner** или **Admin**; целевой пользователь является участником сервера.

**Шаги:**

1. Выбрать участника сервера.
2. Нажать **«Кик»** и подтвердить.

**Ожидаемый результат:**

- Участник удален из сервера.
- Кикнутый пользователь получает событие `guild.force.disconnect` по SignalR - WS-соединение с гильдией разрывается.
- Если пользователь был в голосовом канале - принудительно отключается от него.
- Остальные участники получают событие `member.left` по SignalR.
- Исключенный пользователь может повторно вступить по действующему инвайту.

---

## TC-5.2 Бан участника и немедленное отключение

**FR:** FR-26

**Предусловие:** действие выполняет **Owner** или **Admin**; целевой пользователь является участником сервера и подключен к WebSocket.

**Шаги:**

1. Выбрать участника сервера → **«Забанить»** → подтвердить.
2. Убедиться, что забаненный пользователь получил принудительное отключение.
3. Попробовать принять инвайт на этот сервер от имени забаненного пользователя.

**Ожидаемый результат:**

- Участник удален из сервера и добавлен в список банов.
- Забаненный пользователь немедленно получает событие `guild.force.disconnect` по SignalR.
- Если пользователь был в голосовом канале - принудительно отключается от него.
- Остальные участники получают событие `member.banned` по SignalR.
- Попытка принять инвайт возвращает `HTTP 403 Forbidden`.
- Инвайт-ссылка не позволяет обойти бан.

---

# UC-6: Онлайн-статусы

## TC-6.1 Пользователь отображается онлайн

**FR:** FR-27

**Предусловие:** пользователь авторизован и подключен к WebSocket.

**Шаги:**

1. Пользователь отправляет Heartbeat через SignalR.

**Ожидаемый результат:**

- Статус пользователя отображается как **Online** для участников общих серверов.
- Участники общих серверов получают событие `presence.online` по SignalR (при первом появлении).

## TC-6.2 Пользователь становится офлайн

**FR:** FR-28

**Предусловие:** пользователь авторизован, подключен к WebSocket и ранее отправлял Heartbeat.

**Шаги:**

1. Прекратить отправку Heartbeat (или отключить WebSocket-соединение).

**Ожидаемый результат:**

- Через 30 секунд после последнего Heartbeat статус меняется на **Offline**.
- Участники общих серверов получают событие `presence.offline` по SignalR.

## TC-6.3 Статус остается онлайн при регулярном Heartbeat

**FR:** FR-27

**Предусловие:** пользователь авторизован и подключен к WebSocket.

**Шаги:**

1. Отправлять Heartbeat каждые 20 секунд в течение 60 секунд.

**Ожидаемый результат:**

- Статус пользователя остается **Online** на протяжении всего теста.
