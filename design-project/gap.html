Что нужно достроить, чтобы прототип стал реальной фичей

Документ перечисляет различия между прототипом чата (chat.html) и текущим MVP NexTalk (по README.md и docs/schema.md). Для каждой новой возможности — изменения в БД, REST‑эндпоинты, SignalR‑события и влияние на Outbox / Idempotency.

Документ не описывает реализацию — это inventory работ, по которому можно разложить эпики в трекере, оценить сроки и принять решение «делаем сейчас / откладываем за рамки MVP».

Базовые принципы. Все новые публичные эндпоинты — за JWT (Zitadel sub), идемпотентные POST принимают X‑Idempotency‑Key, новые real‑time события публикуются через Outbox и доставляются клиентам через GatewayEvent SignalR‑hub'а. Любое изменение схемы — миграция в guild или messaging.

01Реакции

NEWM · ~3–5 дн

Кликабельные emoji‑чипы под сообщением. Свои реакции подсвечиваются. Один пользователь — одна реакция на эмодзи на сообщение (toggle).

DB
Новая таблица в схеме messaging: 
messaging.message_reactions (
  message_id  uuid     not null references messaging.messages(id) on delete cascade,
  user_id     text     not null,   -- Zitadel sub
  emoji       text     not null,   -- ":fire:" или unicode
  created_at  timestamptz not null default now(),
  primary key (message_id, user_id, emoji)
);
create index ix_reactions_msg on messaging.message_reactions(message_id);
На message_reactions — каскадное удаление при удалении сообщения.
API
PUT /api/messages/{id}/reactions/{emoji} — поставить (идемпотентно)
DELETE /api/messages/{id}/reactions/{emoji} — снять
В GET /api/channels/{id}/messages добавить агрегацию reactions: [{emoji, count, mine}].
WS
message.reaction.added · message.reaction.removed
payload: { messageId, emoji, userId, count }
Outbox
Reaction insert/delete + outbox_event в одной транзакции — как для message.created. Очерёдность гарантируется таблицей.
Каскады
При message.deleted и guild.deleted — фронт чистит локальный кэш реакций (бэк ничего дополнительно слать не должен благодаря FK).

02Reply‑to (ответ на сообщение)

NEWS · 1–2 дн

Над текстом сообщения — короткий quote оригинала и L‑образный указатель. Клик прокручивает к оригиналу.

DB
Колонка reply_to_message_id uuid null в messaging.messages. FK с on delete set null (если оригинал удалили — quote остаётся как «удалённое сообщение»).
API
В существующий контракт POST /internal/messages (его дёргает WS Gateway) добавить опциональное поле replyToMessageId. В ответах GET /api/channels/{id}/messages денормализованно положить replyTo: { id, authorId, authorName, snippet } для UX без отдельного запроса.
SignalR
В SendMessage на хабе — третий аргумент или DTO с replyToMessageId. message.created payload расширяется тем же replyTo.

03Треды

NEWL · 1.5–2 нед

Отдельная нить ответов под исходным сообщением. В чате видна «footer» плашка N ответов · последний X, при клике открывается панель‑тред (отдельный канал в правом райле).

DB
В messaging.messages: 
thread_root_id  uuid null   -- null = обычное сообщение или корень треда
thread_count    int  not null default 0   -- денорм. счётчик для root
thread_last_at  timestamptz null          -- для сортировки
create index ix_msg_thread_root on messaging.messages(thread_root_id)
  where thread_root_id is not null;
Постит в тред — это insert с thread_root_id = root.id. Счётчик инкрементится триггером или в той же транзакции.
API
POST /api/messages/{rootId}/thread — отправить в тред
GET /api/messages/{rootId}/thread?cursor=… — история треда (та же cursor‑пагинация что и канал)
GET /api/channels/{id}/threads — список активных тредов канала
WS
thread.replied — { rootId, channelId, guildId, count, lastAt }
Подписка: тот же broadcast на гильдию (фронт сам решает, обновлять ли футер).
Зависит от
Reply‑to (если не делать треды поверх reply, отдельная сущность). Outbox‑пайплайн без изменений — это всё те же outbox_event.
Риски
Cursor pagination усложняется (внутри треда отдельная лента). Денорм‑счётчик нужно держать консистентным — лучше всего материализовать триггером.

04Редактирование сообщений

NEWS · 1–2 дн

Автор может изменить текст сообщения. Появляется метка (изменено). Admin/Owner — нет (по соображениям доверия; обсуждаемо).

DB
В messaging.messages: edited_at timestamptz null, edit_count int not null default 0. История версий — не делаем в MVP (можно завести message_revisions позже).
API
PATCH /api/messages/{id} — body { content }. 403 если не автор и не админ. Идемпотентность: по X‑Idempotency‑Key как у POST (PATCH может прийти дважды через retry).
WS
message.edited — { messageId, content, editedAt }
Полезное
На уровне FR — добавить запись об edit в audit log (полезно для модерации; пока есть только delete).

05@‑упоминания

NEWM · 3–5 дн

Inline @username в тексте; адресат получает уведомление с подсветкой соответствующей строки в канале и значок на иконке гильдии. Также @here и @everyone.

DB
Новая таблица: 
messaging.message_mentions (
  message_id  uuid not null references messaging.messages(id) on delete cascade,
  user_id     text not null,   -- получатель (включая @everyone — null)
  kind        smallint not null,  -- 0=user, 1=here, 2=everyone, 3=role
  created_at  timestamptz not null default now(),
  primary key (message_id, user_id, kind)
);
Парсинг — в Messaging Service при INSERT (один проход по тексту, regex @(\\w+) + резолв в Guild через internal call).
API
GET /api/me/mentions?guildId=… — список непрочитанных упоминаний для Inbox (пагинация).
POST /api/messages/{id}/mention-read — пометить как прочитанное.
WS
message.mention — { messageId, guildId, channelId, mentionedUserId, kind }. Доставка — targeted (не всему broadcast'у гильдии, а конкретному user_id через SignalR group "user:{userId}").
Зависит от
В WS Gateway — добавить per‑user группы (присоединять сокет к группе user:{sub} на handshake).

06Закреп сообщений

NEWS · 1–2 дн

Admin/Owner может закрепить сообщение. Закреплённые видны во вкладке «Закреплённое» в правом райле.

DB
В messaging.messages: pinned_at timestamptz null, pinned_by text null. Индекс create index ix_msg_pinned on messaging.messages(channel_id) where pinned_at is not null;
API
PUT /api/messages/{id}/pin
DELETE /api/messages/{id}/pin
GET /api/channels/{id}/pinned
WS
message.pinned · message.unpinned

07Channel topic

EXPANDS · 0.5 дн

Поле «о чём канал» в шапке. Сегодня в схеме у channels есть name и type — нужен ещё topic.

DB
В guild.channels: topic varchar(1024) null
API
Расширить POST /api/guilds/{id}/channels и добавить PATCH /api/guilds/{guildId}/channels/{channelId} с body { name?, topic? }. Права — Owner/Admin (как у создания/удаления каналов).
WS
Расширить существующее channel.created и добавить channel.updated.

08Typing indicator

NEWS · 1 дн

«Х печатает...» снизу чата. Эфемерно (не пишется в БД).

DB
Нет.
SignalR
На хабе — клиентский метод Typing(channelId). Сервер троттлит до 1/сек, шлёт всему broadcast'у гильдии typing.started { channelId, userId, expiresAt } (TTL ~6 сек). Без Outbox — at‑most‑once приемлемо.
Состояние
In‑memory в WS Gateway, ConcurrentDictionary (userId, channelId) → expiresAt. Чистка background‑таймером.

09Unread counts / read state

NEWM · 3–4 дн

Маркер последнего прочитанного сообщения per (user × channel). Считаются непрочитанные и упоминания.

DB
messaging.channel_reads (
  user_id     text not null,
  channel_id  uuid not null references guild.channels(id) on delete cascade,
  last_read_message_id uuid null,
  last_read_at         timestamptz not null,
  mention_count        int not null default 0,
  primary key (user_id, channel_id)
);
API
POST /api/channels/{id}/ack — body { messageId }, сбрасывает unread.
GET /api/me/reads?guildId=… — список (channel, unread, mentions) для иконок и значков.
Подсчёт
SELECT count(*) FROM messages WHERE channel_id = $1 AND id > $last_read_message_id с тем же индексом (channel_id, id DESC) что и у истории. p95 < 50 ms на разумных объёмах.
Связано с
@‑упоминания (mention_count инкрементится тем же триггером, что вставляет в message_mentions).

10Presence: idle / DND / custom status

EXPANDS · 1–2 дн

Сейчас presence — бинарный (online/offline через heartbeat). Прототип показывает online · idle · dnd · offline и кастомный статус‑текст («редактирует Helm chart»).

Хранение
Расширить in‑memory структуру в WS Gateway: 
record UserPresence(
  string Status,        // online|idle|dnd
  string? CustomText,
  DateTime LastHeartbeat
);
В payload presence.online добавить status и customText. Idle — авто, если heartbeat есть, но нет активности в чате > 5 мин (определяется по последнему SendMessage).
SignalR
Клиентский метод SetPresence(status, customText). Сервер пушит presence.updated в broadcast гильдиям пользователя.
Без БД
В MVP кастомный статус — эфемерный (сбрасывается при реконнекте). В follow‑up — таблица guild.user_settings(user_id, custom_status).

11Slash‑команды

NEWM · 3–5 дн

Меню по символу / в композере. Большая часть — клиентские шорткаты (вставка форматирования, выбор GIF). Несколько серверных: /poll, /remind, /incident, /911.

Архитектура
Команды передаются как обычные сообщения через SendMessage, помеченные флагом isCommand в DTO. Messaging Service маршрутизирует на хендлер по cmd[0]. Серверный ответ — system‑сообщение (см. #14).
DB
Только для /poll и /remind: messaging.polls(message_id, options[], votes), messaging.reminders(user_id, channel_id, fire_at, text).
Фоновое
Для /remind — новый BackgroundService‑тикер (как Outbox), полит каждые 10 сек.

12Поиск по сообщениям

NEWL · 1.5 нед

Глобальный поиск с фильтрами (канал, автор, период). В прототипе — выпадающая панель справа с подсветкой совпадений.

DB
PostgreSQL FTS на колонке content: 
alter table messaging.messages
  add column tsv tsvector
  generated always as (to_tsvector('russian', content)) stored;

create index ix_msg_tsv on messaging.messages using gin (tsv);
API
GET /api/search/messages?q=&channelId=&authorId=&from=&to=&cursor=
Ранжирование ts_rank_cd; в payload — выделенные совпадения (ts_headline).
Авторизация
Важно: поиск ограничен каналами, к которым у user'а есть доступ. Для каждой страницы результата — internal‑call в Guild /internal/channels/{id}/access для фильтрации, либо джойн с guild.members. Без этого можно случайно вернуть сообщения из чужой гильдии.
Риски
На больших объёмах russian dictionary — ~30% места на индекс. Подумать про шардирование по guild_id в долгосрочной перспективе.

13Inbox / уведомления

NEWL · 1–1.5 нед

Сводная лента: упоминания, ответы в ваших тредах, реакции на ваши сообщения. Со статусом «прочитано/нет» и группировкой по каналам.

DB
messaging.notifications (
  id          uuid pk default gen_random_uuid(),
  user_id     text not null,        -- получатель
  kind        smallint not null,    -- 0=mention, 1=thread_reply, 2=reaction, 3=system
  source_id   uuid not null,        -- message_id или thread_id
  guild_id    uuid not null,
  channel_id  uuid not null,
  read_at     timestamptz null,
  created_at  timestamptz not null default now()
);
create index ix_notif_user on messaging.notifications(user_id, read_at, created_at desc);
Генерируется триггерами при mention/reaction/thread_reply, либо записями в outbox.
API
GET /api/me/inbox?status=unread&cursor=
POST /api/me/inbox/{id}/ack
POST /api/me/inbox/ack-all
WS
inbox.added — пуш на per‑user группу. Клиент сам ставит badge на иконку сервера/канала.
Зависит от
@‑упоминания (#5), реакции (#1), треды (#3) — Inbox их агрегирует. Имеет смысл делать после них.

14Bot / system‑сообщения

NEWM · 3–4 дн

Сообщения, авторами которых являются сервисы, а не люди: deploy‑бот, реакция на /incident, уведомления о бане, результаты опроса.

DB
В messaging.messages: author_kind smallint not null default 0 (0=user, 1=bot, 2=system). У system — author_id допускается magic‑UUID 00000000‑0000‑0000‑0000‑000000000001.
API
POST /internal/messages/system (только internal — для outbox‑консьюмеров guild service, напр. при бане). Для внешних ботов — отдельные машинные JWT с ограниченным scope (Zitadel «application user»), которые могут писать в каналы где у них есть Member.
UI
В message.created payload — поле authorKind. Фронт рисует BOT / SYSTEM badge и отдельный стиль (см. прототип, release‑bot).

15Profile hover card

EXPANDS · 1 дн

При hover на аватаре в чате/райле — карточка с именем, ролью, статусом, activity, кнопками «упомянуть» / «отправить DM» (DM пока отключено).

Бэкенд
Дополнительных эндпоинтов не требуется — используется существующий GET /api/guilds/{id}/members (список с ролями), который уже вызывается при открытии гильдии. Активити берётся из presence (см. #10).
Фронт
Только React‑компонент + portal, на 200 строк включая стили.

16Вложения и превью ссылок

BLOCKEDL · 2+ нед

Изображения и файлы в сообщениях, превью ссылок (CI bot, дашборды). В README явно за рамками MVP — нужен MinIO и пайплайн.

Инфра
MinIO (S3‑совместимое) + bucket policy + presigned URLs. В docker‑compose и Helm‑chart — новый сервис. Для link‑unfurl — отдельный worker, опрашивающий outbox новых сообщений и кэширующий OG‑метаданные.
DB
messaging.attachments (id, message_id, kind, url, mime, size, width, height) и messaging.link_previews (url pk, og_title, og_image, og_desc, fetched_at).
API
POST /api/uploads/presign — выдать presigned URL, клиент грузит напрямую в S3.
POST /api/messages — в DTO список attachmentIds.
Решение
Не включать в Wave 1‑3. Картинки и файлы — отдельный эпик, его удобно делать сразу с аватарками (которые также за рамками MVP).

17Voice: VAD / камера / демонстрация экрана

EXPANDM · 3–5 дн

В прототипе — обводка говорящего, waveform, иконки камеры/screen share. VAD в README отмечен как Should (FR‑24) — это всё клиентская логика livekit‑client, бэк не трогаем.

Бэкенд
Voice Service не меняем — он выдаёт JWT для LiveKit с claims, разрешающими публикацию audio. Чтобы разрешить video/screen — добавить grants: canPublish=true, canPublishSources=[camera, screen_share] в LiveKit access token (только если фича включена для канала).
Фронт
Подписаться на activeSpeakers event у Room. Добавить кнопки в VoiceControls для localParticipant.setCameraEnabled / setScreenShareEnabled.
UX
Опционально: добавить чип к каналу «v‑share активен» — нужно отдельное событие voice.screen_share.started, которое триггерится по participant track published. Можно отложить.

Дорожная карта

Группировка по очередности, исходя из стоимости/ценности и зависимостей. Числа — порядок, в идеале — не строгий.
